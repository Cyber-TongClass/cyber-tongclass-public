# OA 我的提交与审批详情设计

## 目标

将成员端 OA 的历史提交从单个表单页中的记录表升级为独立的“我的提交”列表和详情页。列表中的每条记录以“`表单名的第 N 次提交`”命名。详情页展示提交瞬间的紧凑版表单内容，以及采用飞书阅读模式的单级审批时间线。

本次采用方案 A：统一汇总全部 OA 表单提交。审批流程限定为单级：`提交 → 管理员审核 → 结束`。不新增多级、可配置审批节点、抄送或自动通过规则。

## 用户体验

在 OA 入口页提供“我的提交”入口，跳转到独立列表页。列表按提交时间倒序，逐行显示提交标题、提交时间和当前审核状态。标题中的 N 按同一表单下该用户的提交顺序计算：最早一次为第 1 次提交，即使列表以最新在前显示，编号仍稳定。

每条记录链接到独立详情页。详情页显示表单名称、该次提交编号、状态和提交时间。表单内容使用两列的紧凑只读布局；文件答案保留受权限保护的下载链接，表格答案按列呈现，不输出 JSON。

“审批记录”使用纵向时间线：绿色已完成圆点表示提交和完成的审核，橙色圆点表示待审核，灰色圆点表示未开始/结束状态。提交节点显示提交人和提交时间；审核节点在待审核时显示“等待管理员处理”，在已处理时显示审核人、审核决定、审核时间与管理员备注；结束节点根据审核结果显示“已通过”“已拒绝”或“需补材料”。

原表单页不再承担历史记录浏览：保留新建提交和（在允许时）修改自己的提交；“我的提交记录”替换为前往统一列表的入口，避免重复且让所有历史记录具有稳定链接。

## 数据与接口

每一份 `oaFormSubmissions` 新增 `formSnapshot`。它在提交创建时写入，包含表单标题、描述、字段定义、结果字段定义及结果可见性。这一快照是详情渲染的唯一表单结构来源，确保管理员之后修改表单不会改变历史提交的展示。已有提交没有快照时，以当前关联表单构建只读回退快照，保证旧数据仍可查看。

`listMine` 的每个返回项补充安全的 `formTitle`（优先快照标题）以及可选的快照回退数据，供统一列表计算标题。新增“获取我自己的单个提交”的查询，仅允许提交人访问，并返回被提交人可见性规则过滤后的结果字段。前端 API 封装保持在 `src/lib/api.ts`，组件不直接调用 Convex。

为支持深链接详情，新查询按提交 ID 读取并验证提交所有权。附件仍沿用现有 `getAttachmentUrl` 的所有权校验，不改变下载权限。

## 前端结构

新增 `src/app/intranet/forms/submissions/page.tsx` 作为统一“我的提交”列表页，新增 `src/app/intranet/forms/submissions/[id]/page.tsx` 作为详情页。将紧凑答案渲染、提交标题及审批时间线拆入 OA 组件目录，避免把格式化逻辑复制到路由页面。

提交次数编号由同一 `formId` 的全部本人提交按 `submittedAt` 升序排序后计算；若时间相同，用提交 ID 作为确定性的次级排序。该逻辑以纯函数实现并覆盖测试。紧凑答案渲染明确覆盖普通字段、多选、附件和表格字段。

## 错误与兼容

列表和详情分别提供加载、空状态、记录不存在/无权限状态。缺失快照的历史记录使用关联表单回退；关联表单也不存在时，仍显示提交基础信息和审批记录，表单字段区提示历史表单不可用。

状态 `pending`、`approved`、`rejected`、`needs_changes` 均有完整的时间线与文案。管理员备注为空时不渲染空白备注块。

## 验证

在实现前先为纯提交编号和审批时间线数据构建函数添加失败的单元测试，再最小化实现使其通过。随后运行相关测试及 `npm run lint`。还应以本地开发环境手动核验：同表单多次提交的编号、不同表单混合排序、快照表单字段更新后的历史详情、四种审批状态、文件链接与表格字段展示。
